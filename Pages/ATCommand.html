AT Command
Automation

No need to memorise AT commands,
This app simply works from a text file.


Background Story

I created this app to rectify the memorising AT commands issue at work. Probably memorising AT commands is one of my least favourite things at work, so I designed this app to help me remember all the AT commands that I need.
The main purpose of creating this app is to simplify communication and testing procedures of the boards at work. These boards could connect to a computer using a USB to RS232 converter and communicate the status of the different features of the board. The features of the board could be activated or deactivated using this app. I made this app over two weekends at home and used it at work.

Key Features

This app is similar to a serial terminal (e.g. putty) with in-built memory of AT commands, a tab-delimited text file captures all the commands. The text file can be updated with the new AT commands, this feature allows the user to add or remove commands without any need to change the back-end code of the app.
The app is designed in LabVIEW and has two functional modes, "Serial Mode" and "Demo Mode".
The serial mode allows the user to connect a USB to RS232 converter to the computer and communicate with a board. On the other hand, the demo mode is only for demonstration of the app feature without communicating with a board. The demo mode is a standalone mode that works without any extra hardware other than a computer.
The user can configure the critical aspects of the communication such as Baudrate, Carriage Return, and Line Feed. These features can simply adjusted according to the board specifications.
All the messages from the board are received and shown in the "Received" terminal. The drop-down menus and buttons are used to create the specific AT command. Once these commands are created, the "sent" field in the app shows the actual command passed to the board.

Challenges and Solutions

I did this project at home but meant to use it at work. Not having access to the hardware and not knowing the response of the board caused so much difficulty in testing and verifying the back-end code. I developed the basics of the app in demo mode however for the serial mode I needed to have at least one RS232 to ensure that my code didn't crash and detect the presence of the RS232 converter.
The below screenshot shows the pop-up warning that allows the user to select the mode.


The serial communication was also a bit challenging because it took a bit of time to configure the timeouts and delays to ensure that all the data frames were received properly and there was no packet drop/loss.
I managed to find a suitable buffer size for the received characters by testing the communication between two RS232 converters. To confirm that the commands are sent and received in my code, I used two RS232 converters and attached them to act as sender and receiver of AT commands. I sent commands using one of them and the second one was observed for the messages via putty (serial monitor).

Final Words

This app was my quickest LabVIEW design. It has several useful features and opportunities for further development. One of the features that could be added to the app is automatic testing. Potentially, the user could set an interval to run all the commands one by one and capture the results in a spreadsheet or any database.
The next step for this project could be adding extra indicators and gauges to represent the analog readings rather than returning just a raw number.
All in all, I learned more about RS232 and how to buffer, decode, and analyse messages in LabVIEW.
